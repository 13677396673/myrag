## 一、项目概述

### 1.1 项目目标

构建一个**生产级可用的 RAG（Retrieval-Augmented Generation）智能知识库系统**，支持多格式文档的上传、解析、向量化存储、语义检索与 LLM 智能问答，具备专业 Web UI 与 RESTful API。

项目定位为：
- **学习 RAG 原理的实践项目** — 跑通完整 RAG Pipeline
- **简历上展示完整工程能力的作品** — 前后端分离、模块化设计、可扩展架构
- **可实际部署使用的工具** — Docker 部署、云服务器可访问

### 1.2 核心能力

- 上传多种格式文档 → 自动解析与切片 → 向量化存储 → 用户提问 → 语义检索相关片段 → LLM 生成回答
- 支持对话历史上下文
- 支持多用户与文档空间隔离
- 支持纯本地部署（Windows 10+）及云端部署（Linux 服务器）

### 1.3 目标用户

| 角色 | 说明 |
|------|------|
| **普通用户** | 知识库使用者，通过 Web UI 上传文档并提问 |
| **管理员** | 管理用户、文档、系统配置 |

### 1.4 项目非功能性目标

| 维度 | 目标 |
|------|------|
| **可扩展性** | 所有核心模块基于抽象接口设计，支持插件式替换 |
| **可维护性** | 代码结构清晰，模块解耦，详细注释 |
| **兼容性** | Windows 10+ 本地开发运行 + Linux 服务器部署 |
| **易用性** | Docker 一键部署；开发环境最小依赖即可启动 |
| **安全性** | JWT 鉴权；用户数据隔离 |

### 3.2 RAG 核心组件选型

#### 3.2.1 大语言模型（LLM）— 混合架构

**设计原则**：通过抽象 `LLMBackend` 接口支持多种 LLM 后端，配置化切换。

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **DeepSeek API** ✅ 默认 | 价格极低（≈OpenAI 1/20）、中文好、质量高 | 需要网络 | 日常开发与使用，性价比最高 |
| **OpenAI API** | 质量极高、生态最成熟 | 费用较高、需要网络 | 备选/对比测试 |
| **Ollama（本地）** | 免费、数据不出本机、不依赖网络 | 4GB 内存只能跑 1B-3B 小模型 | 完全离线场景 |
| **通义千问 API** | 中文友好、有免费额度 | 质量略逊于 GPT-4 | 备用方案 |

**推荐策略**：
- **开发/日常**：DeepSeek API（极低成本 + 高质量）
- **纯本地**：Ollama + Qwen2.5 / LLaMA 系列小模型
- **扩展预留**：OpenAI 兼容接口规范，可对接任意 OpenAI 兼容 API

#### 3.2.2 Embedding 模型

**设计原则**：通过抽象 `EmbeddingBackend` 接口支持多种方案。

| 方案 | 优点 | 缺点 |
|------|------|------|
| **BAAI/bge-small-zh-v1.5** ✅ 默认 | 免费、384维速度快、中文好、轻量（~200-400MB） | 中文外场景略弱 |
| **text2vec-base-chinese** | 中文语义理解好 | 维度较高、体积稍大 |
| **OpenAI text-embedding-3-small** | 质量高、零本地资源 | 有费用、需要网络 |
| **DeepSeek Embedding** | 价格低、质量好 | 需要网络 |

**推荐**：默认使用 `BAAI/bge-small-zh-v1.5` 本地运行（约 200-400MB 内存），同时支持 API 方式通过配置切换。

#### 3.2.3 向量数据库

**设计原则**：通过抽象 `VectorStore` 接口支持多种后端，一键切换。

| 方案 | 优点 | 缺点 | 阶段 |
|------|------|------|------|
| **ChromaDB** ✅ MVP | 零配置、嵌入式、纯 Python 运行、文件存储 | 不适合大规模生产 | MVP 首选 |
| **FAISS** | 轻量极快、纯本地 | 缺乏持久化与过滤 | 可搭配 Chroma 使用 |
| **Milvus（pymilvus-lite）** | 轻量嵌入版、生产版可扩展 | 学习曲线稍陡 | 后续扩展 |
| **PGVector** | 统一数据存储、支持 SQL 过滤 | 需要 PostgreSQL | 云部署首选 |
| **Qdrant** | 性能好、支持过滤、REST API | 需要额外服务 | 后续扩展 |

**推荐策略**：
- **MVP 阶段**：ChromaDB（最快上手，零配置）
- **生产部署**：PGVector（一套 PostgreSQL 搞定关系库 + 向量库）
- **扩展预留**：统一 `VectorStore` 接口

#### 3.2.4 前端方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **React + Vite + Ant Design** ✅ | 组件丰富、社区庞大、简历加分 | 需要 Node.js 环境 |
| React + Next.js | SSR、全栈框架 | 项目复杂度提升 |
| Vue 3 + Element Plus | 中文文档好 | 生态略小于 React |

**推荐**：React 18 + TypeScript + Vite + Ant Design 5.x

### 3.4 核心抽象接口设计

所有核心组件均通过抽象接口定义，确保可扩展性。

```
LLMBackend          ←  DeepSeekAPI, OpenAIAPI, OllamaLLM, 自定义...
  → generate(prompt, context) → str
  → generate_stream(prompt, context) → AsyncIterator[str]

VectorStore         ←  ChromaDBStore, FAISSStore, MilvusStore, PGVectorStore, 自定义...
  → add_embeddings(ids, vectors, metadatas)
  → search(query_vector, top_k, filter) → List[SearchResult]
  → delete(ids)
  → health_check() → bool

EmbeddingBackend    ←  BGESmallEmbedding, OpenAIEmbedding, DeepSeekEmbedding, 自定义...
  → embed_text(text) → List[float]
  → embed_documents(texts) → List[List[float]]

DocumentParser      ←  PDFParser, DocxParser, PPTXParser, OCRParser, WebParser, 自定义...
  → parse(file_path) → List[DocumentChunk]
  → supported_extensions() → List[str]

TextSplitter        ←  FixedSizeSplitter, SemanticSplitter, MarkdownSplitter, 自定义...
  → split(text, metadata) → List[DocumentChunk]

Retriever           ←  VectorRetriever, BM25Retriever, HybridRetriever, 自定义...
  → retrieve(query, top_k, filter) → List[SearchResult]
```

### 7.2 向量数据库方案对比

| 维度 | ChromaDB | FAISS | Milvus | PGVector |
|------|---------|-------|--------|----------|
| 部署难度 | ⭐（零配置） | ⭐（轻量） | ⭐⭐⭐ | ⭐⭐ |
| 检索性能 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
